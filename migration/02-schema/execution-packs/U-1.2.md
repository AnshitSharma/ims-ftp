## Database Changes (1 seeder + rollback)
2026_07_06_002_create-config-resources.sql:
```sql
CREATE TABLE config_resources (
  id BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT,
  config_uuid CHAR(36) NOT NULL,   -- same collation rule as U-1.1
  resource ENUM('cpu_socket','dimm_slot','pcie_slot','pcie_lane','m2_slot','u2_slot',
                'drive_bay_2_5','drive_bay_3_5','sfp_port','psu_watt','riser_slot') NOT NULL,
  provider_id BIGINT UNSIGNED NOT NULL,
  slot_ref VARCHAR(64) NULL,
  capacity INT NOT NULL,
  consumer_id BIGINT UNSIGNED NULL,
  UNIQUE KEY uq_discrete (config_uuid, resource, slot_ref),
  KEY k_provider (provider_id), KEY k_consumer (consumer_id),
  CONSTRAINT fk_cr_config FOREIGN KEY (config_uuid) REFERENCES server_configurations(config_uuid),
  CONSTRAINT fk_cr_provider FOREIGN KEY (provider_id) REFERENCES config_components(id) ON DELETE CASCADE,
  CONSTRAINT fk_cr_consumer FOREIGN KEY (consumer_id) REFERENCES config_components(id) ON DELETE RESTRICT
) ENGINE=InnoDB;
```
ON DELETE RESTRICT on consumer_id is the DB-level dependency guard: a provider row cannot vanish
while consumed (INV-1 family). Scalar resources (pcie_lane, psu_watt): slot_ref NULL, capacity=N,
consumption modeled as separate rows with negative? NO — consumption of scalars = rows with
provider_id of provider, consumer_id set, capacity = amount consumed (positive), and providers have
consumer_id NULL. ledger_report sums: consumed(resource) ≤ provided(resource). Document in header.
