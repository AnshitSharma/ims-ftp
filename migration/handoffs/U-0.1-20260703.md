# Handoff — U-0.1 — 2026-07-03

## Current State
`ServerBuilder::addComponent()` is now fail-closed end to end: every catch block on the add
path (CPU pre-check, RAM pre-check, comprehensive compatibility check, and the PCIe/NIC/HBA
slot-assignment step) rolls back its own transaction and returns `success=false` instead of
logging and silently continuing. The advisory pre-check in `server_api.php`'s
`handleAddComponent` does the same — a thrown validation exception now responds HTTP 500
"Validation unavailable - request rejected (fail-closed)" instead of letting the add proceed
with a fake "unavailable" notice. Repo-wide `grep -rn "Continue without" core/ api/` is clean.

  correct defense-in-depth per INV-5's literal text, but are effectively unreachable dead code
  right now given current layering — flagging so nobody assumes my regression test exercises
  those exact 3 lines. It exercises `addComponent()`'s overall fail-closed contract end-to-end
  (landing in the pre-existing, already-correct Phase-3 JSON-existence catch) plus a static
  string check for the literal fix.

  This unit's changes are uncommitted as of writing this handoff — see Next Prompt.
