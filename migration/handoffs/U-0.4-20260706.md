  and NULL ServerUUID, (2) no inventory row referenced by any `server_configurations` JSON

- `scripts/verify/orphan_report.php` (new): shells out to `scripts/audit-orphans.php` (dry-run,

  `tests/fixture_scenarios_real.php` (same UUIDs, same JSON shapes, same
  `validateComponentAddition` / `validateConfiguration` + `validateConfigurationEnhanced` calls)
  but connects via `GOLDEN_DB_*` env vars instead of a hardcoded production DB name, so it can run
  against any seeded scratch DB. `--capture-baseline` writes `reports/perf-baseline.json`; default
  mode compares current p95 against it (green iff delta ≤ +20%) and writes
  `reports/performance-<ts>.json`.
- `scripts/verify/run_all.php` (new): `REGISTRY` const (report → script → available/lands_in) and
  `GATE_REPORTS` const (copied verbatim from each phase's `gate_reports` in

  (schema/equivalence print SKIPPED); `--gate P<N>` runs exactly that phase's `gate_reports`
  (`all` for P10 expands to every registry key). Unavailable reports print
  `<name>: SKIPPED (lands in <unit>)` and never affect exit code — the registry's `available`
  flag is the only thing that can flip a report from SKIPPED to counted, so a SKIPPED report can
  never green-wash a gate.

- `reports/perf-baseline.json` (new, committed per the gitignore's exception): see Known Risks —

## Known Risks
- **Same environment gap as U-0.3's handoff**: this remote/cloud session has no MySQL/MariaDB
  server, no `ims-data/`, and no production DB dump preinstalled. To verify anything at all, this
  session again installed `mariadb-server-core` via apt, ran `mariadbd` manually against a scratch
  datadir, and hand-built a minimal schema (`server_configurations`, the 7 inventory tables the
  R1-R10 scenarios touch, `server_configuration_history`). This does not persist outside the
  container.
- **`reports/perf-baseline.json` is NOT a real production-catalog baseline.** Without `ims-data/`,
  every `validateComponentAddition`/`validateConfiguration` call in the R1-R10 replay fails
  internally (caught by `ServerBuilder`, logged via `error_log`, returns `success=false` rather
  than throwing) — so the captured baseline (`add`: p50=0.207ms/p95=0.521ms, n=8;
  `finalize`: p50=0.582ms/p95=1.174ms, n=2) measures the cost of the DB round-trip +
  fail-fast error path, not real validation work. Per the Human Verification Checklist item ("perf
  baseline captured AFTER U-0.1..0.3, guards included"), **a human/verify session with real
  `ims-data/` + a production dump should re-run
  `php scripts/verify/performance_report.php --capture-baseline` against a properly seeded
  scratch DB before this baseline is trusted for the +20% regression gate** in later units
  (P6/P8 both list `performance` in their `gate_reports`).
- **`baseline` and `regression` gate-report names** (used in P0/P1/PL/P3/P4/P5/P6/P7/P9's
  `gate_reports`) have no dedicated `scripts/verify/*.php` script — the 9 report specs in
  `11-verification/README.md` don't define one for either. They're registered in `run_all.php`'s
  `REGISTRY` as permanently `available: false`, pointing at the existing
  `tests/characterize_compatibility.php` and `tests/regression/*.php` respectively, since those
  are what those two gate-report names conceptually mean. If a later unit is meant to wrap either
  of those in a real `run_all.php`-integrated script, that wasn't specified anywhere I could find
  in `migration/` — flagging for a human to confirm this reading is correct, since as it stands
  **no gate whose `gate_reports` list only contains `baseline`/`regression`/already-SKIPPED names
  can ever go fully GREEN via `run_all.php`** (it will always show SKIPPED lines for those, never
  GREEN) — gate-opening for those phases will need to fold in `characterize_compatibility.php` and
  the regression suite by hand until/unless a report script is written for them.
- **INV-11 box**: this unit created 6 files (4 scripts + `.gitignore` + `perf-baseline.json`),
  ~639 LOC across the 4 PHP scripts — over the generic "≤500 new/modified LOC" default in
  `ARCHITECTURAL_INVARIANTS.md`. The pack's own box statement ("Files Created (5) / Modified (0)")
  restates file count but not a LOC ceiling (unlike U-0.3's pack, which stated one explicitly),
  and the five outputs it lists are only buildable together as one cohesive harness — I did not
  find a sensible split point without leaving `run_all.php` unable to reference reports that don't
  exist yet. Flagging this honestly rather than silently exceeding the general cap unremarked;
  a human should confirm this is an acceptable exception for an "observability bootstrap" unit.
