## This unit directly resolves (in code) the question flagged in U-1.1
The pack's `insert(string $configUuid, array $row, int $actor): int` signature implies a literal
INSERT. But per U-1.1's schema decision (`uq_inventory_once` on `(inventory_table, inventory_id)`,
no `removed_at`), a physical unit that was placed then removed already has a row (tombstoned).
A bare `INSERT` would hit that unique key and fail with a duplicate-key error on the very first
remove-then-re-add — a real correctness bug, not a hypothetical.

**Resolution implemented**: `insert()` uses `INSERT ... ON DUPLICATE KEY UPDATE` against
`uq_inventory_once`. A genuinely new physical unit gets a fresh row (normal INSERT path); a
previously-tombstoned one is reactivated in place (`removed_at` cleared, `config_uuid`/`slot_ref`/
`serial_number`/`added_at`/`added_by` rewritten to the new placement, same `id` reused). Callers
see one `insert(...)` call either way — which SQL path actually ran is an internal detail, not
exposed in the return value's meaning (it's `config_components.id` in both cases).

This means **the interpretation from U-1.1 is now load-bearing in actual code**, not just a schema
comment. The unit test (`tests/unit/config_component_repository_test.php`) explicitly proves the
reactivation path: insert → tombstone → insert again → asserts the SAME row `id` is returned and
`liveRows()` shows exactly one row, not a duplicate-key exception. **If a human reviewing
U-1.1's flagged decision decides the OTHER interpretation was intended (append-only, 3-column key
with `removed_at`), this file's `insert()` method must change too** — the `ON DUPLICATE KEY UPDATE`
clause only makes sense against the 2-column key this session chose.

I did not attempt to re-ask via `AskUserQuestion` this session (both U-1.1 attempts failed with a
transport error, logged there) — proceeded on the same interpretation for consistency across the
now-3 units that touch it (U-1.1 schema, this unit's code, and implicitly whatever U-1.5/dual-write
will do).
