## Design decision: `actor` defaults to `0`
Neither `addComponent()` nor `removeComponent()` has any authenticated-user context in scope today
— `ServerBuilder` is never constructed with a user id, and no caller passes one in. Rather than
plumb a new parameter through both public method signatures (a real, unbounded-scope change, and
outside this unit's 5-file box) for a flag that is off by default and whose `actor` column is
explicitly `DEFAULT 0` in the U-1.3 schema for exactly this reason, both hook call sites pass a
literal `0`. This is a best-effort audit field, not correctness-critical (matches the pack's own
"best-effort NULL" framing for `parent_id`). If real actor attribution becomes load-bearing before
`COMMAND_LAYER_ENABLED` lands (which will own passing a real actor through), a follow-up unit should
thread it through — not urgent, flagging for visibility only.
