## Flagged for a human/verify session to confirm
1. **`RISER_SUBTYPE_KEYS` is a best-effort guess, not confirmed.** The pack's normalization rule
   ("legacy 'pciecard' with subtype Riser Card ⇒ compare as 'riser'") names a JSON field this unit
   was not permitted to discover — its "Files To Read" scoped `ServerBuilder.php` to lines 61-120
   only (`extractComponentsFromJson`'s cpu/ram/storage/caddy blocks), which don't reach the pciecard
   block (further down the same function). `equivalence_report.php` checks
   `$entry['subtype']`/`$entry['card_type']`/`$entry['type']` case-insensitively for the substring
   "riser" as a placeholder. **Also note**: today, `ConfigComponentWriter` (U-1.5) always forwards
   whatever `$componentType` `ServerBuilder::addComponent()` was called with, which per
   `componentTables` (line ~19-29) is always literally `'pciecard'` — there is no `'riser'` entry
   point yet at the ServerBuilder dispatch level, even though `config_components.component_type`'s
   ENUM already includes `'riser'` as a value (from U-1.1's schema, presumably anticipating a later
   unit, maybe U-B.2, actually splitting them out). Until something writes `'riser'` to the row
   side, a JSON-side pciecard-riser normalized to `'riser'` will NOT match a row-side plain
   `'pciecard'` entry — this normalization only pays off once whatever produces `'riser'` rows
   exists. Confirm the real key name against `ims-data/pciecard/*.json` and re-verify this
   normalization once that's known.
2. **`slot_ref` extraction on the JSON side is also best-effort** (`$entry['slot_position'] ??
   $entry['slot_ref'] ?? $entry['slot_id'] ?? null`), for the same reason — the allowed read range
   didn't reach the pciecard/nic/hbacard blocks where slot info would actually be written into the
   JSON. `cpu`/`ram`/`storage`/`caddy` (the only blocks this unit could read) carry no slot concept,
   so the acceptance-test fixture (built with `cpu`, see below) can't exercise this key at all.
3. **A genuine pre-existing legacy-code asymmetry was discovered while building the acceptance
   fixture, not introduced by this unit**: `extractComponentsFromJson()`'s `ram` block (lines 84-97,
   within this unit's allowed read range) never extracts `serial_number` from the RAM JSON entry,
   unlike its `cpu` block (lines 68-79), which does. Building the "dual-written fixture, exit 0"
   acceptance test with `component_type='ram'` reliably produced a false-positive diff (row side
   has a real serial, JSON side canonicalizes to `null` because the extraction drops it) — this is
   real, not a bug in this report. Switched the acceptance fixture to `cpu` (which round-trips
   serial_number correctly) to get a true GREEN. **This means, right now, `equivalence_report.php`
   will always flag every dual-written RAM entry with a serial number as non-equivalent** (JSON
   side has serial=null, row side has the real serial) until either `extractComponentsFromJson()`'s
   ram block is fixed to include `serial_number` (a legacy-code fix, out of this unit's scope) or a
   RAM-specific normalization is added here to compensate. Flagging rather than silently patching
   around it, since "fix root causes, not symptoms" per CLAUDE.md and this isn't this unit's box.

## Bug fixed during this unit (in the new file, not legacy code)
The `--self-test` block originally called `exit()` inside the `try` of a `try { ... } finally { ...
cleanup... }` — in PHP, `exit()`/`die()` inside a `try` does **not** run `finally`, so the
self-test's throwaway config leaked on every run (confirmed: it showed up in a subsequent `--all`
scan as a spurious diff). Fixed by moving `exit()` calls after the `try/finally` completes, using a
`$caught` flag captured inside. Also switched the self-test's induced mismatch from a
serial-number difference (entangled with the ram quirk above) to a deliberately different
`spec_uuid` on each side — an unambiguous mismatch independent of any per-type extraction quirk.
