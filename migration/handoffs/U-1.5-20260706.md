## Why this is blocked
The pack (`migration/02-schema/execution-packs/U-1.5.md`) says, for the `afterLegacyAdd` hook:
> Inventory id/table: reuse `$componentDetails['ID']` + `getComponentInventoryTable()`

`$componentDetails` in `ServerBuilder::addComponent()` is set at line 643:
```php
$componentDetails = $lockResult['data'];
```
`$lockResult` comes from `lockAndCheckComponent()` (`ServerBuilder.php:5290-5325`), whose query is:
```sql
SELECT UUID, SerialNumber, Status, ServerUUID, Location, RackPosition
FROM `$table`
WHERE UUID = ? [AND SerialNumber = ?]
[ORDER BY Status ASC]
FOR UPDATE
```
**There is no `ID` column in that SELECT.** `$componentDetails['ID']` does not exist anywhere in
the current code — I grepped the whole file for any numeric-PK read on an inventory table
(`SELECT ... id ... FROM ...inventory`, `['ID']`, `` `ID` ``) and found nothing. Every inventory
lookup in this codebase keys by `(UUID, SerialNumber)`, never a numeric primary key.

This could mean one of two things, and I can't tell which from what's in this repo:
1. The ten `{type}inventory` tables DO have a standard auto-increment `id`/`ID` primary key at the
   schema level, but the application has simply never needed to SELECT it — in which case the fix
   is a one-line addition to `lockAndCheckComponent()`'s SELECT list. Safe and additive (nothing
   currently reads that key, so adding it breaks nothing) — but it's a **third** ServerBuilder.php
   modification site, when the pack's box explicitly says "exactly two call sites" (the two hook
   points) and "Files Modified: 1" as part of a 5-file box (2 created + 1 modified = 3, per the
   pack's own accounting for `dual_write_test.php`).
2. The tables genuinely have no separate numeric PK (some schemas key purely on UUID), in which
   case `config_components.inventory_id BIGINT UNSIGNED` (from U-1.1's already-shipped seeder) has
   no value to populate at all, and either that column's design needs revisiting or `inventory_id`
   needs a different definition (e.g. the UUID itself, or a composite key) — a schema-level
   question, not something to patch around inside `ServerBuilder.php`.

I have no access to the real production schema in this sandboxed session (same environment gap as
every prior unit: no MySQL, no production dump, no `docs/architecture/DATABASE_SCHEMA.md` — that
doc is listed in CLAUDE.md as local-only/gitignored and isn't present in this checkout) to tell
which case applies. Guessing here risks either quietly exceeding the pack's file-box, or writing
against a column that may not exist — and this is happening on `ServerBuilder.php`, the file the
whole project treats with the most caution (CLAUDE.md: "NEVER load it whole; packs give line
ranges"), while an already-unconfirmed design decision from U-1.1 is still open. Per
`ARCHITECTURAL_INVARIANTS.md`'s explicit protocol ("If an instruction in a pack appears to conflict
with [reality] ... STOP. Do not implement.") and `SESSION_PROTOCOL.md` Step 2.3 ("If a listed
file/range does not match the pack's description ... STOP ⇒ handoff status blocked"), stopping
here rather than guessing a third time in a row.
