## ‚öÝÔ∏è Flagged design decision ‚Äî needs human confirmation
**The pack's literal DDL conflicts with INV-1 as written, and I resolved it without waiting for
sign-off** (I attempted to ask via `AskUserQuestion` twice; the tool errored both times with
"stream closed before response received" ‚Äî logged here so it's visible, not silently skipped).

The pack specifies:
```sql
UNIQUE KEY uq_inventory_once (inventory_table, inventory_id, removed_at)
```
INV-1's own mechanical check (`ARCHITECTURAL_INVARIANTS.md`) is:
```sql
SELECT inventory_table, inventory_id, COUNT(*) c FROM config_components GROUP BY 1,2 HAVING c > 1; -- must return 0 rows
```
This check has **no `removed_at` filter** ‚Äî it demands at most one row ever, full stop. But MySQL/
MariaDB treat `NULL` as distinct in unique keys, so the pack's 3-column key does NOT stop two
simultaneous live rows (`removed_at IS NULL`) for the same physical unit, and it explicitly
permits a second INSERT once the first row is tombstoned (component removed then re-added) ‚Äî
which would leave 2 total rows for that `(inventory_table, inventory_id)`, failing INV-1's check
as literally written.

**What I implemented instead** (seeder header has the full reasoning):
```sql
UNIQUE KEY uq_inventory_once (inventory_table, inventory_id)   -- no removed_at
```
Exactly one row ever per physical unit ‚Äî matches INV-1's check verbatim. The tombstone lifecycle
changes from "INSERT a new row on remove, INSERT again on re-add" to "the SAME row is UPDATEd":
`removed_at = NOW()` on removal, `removed_at = NULL` + `config_uuid`/`slot_ref`/`serial_number`/
`added_at` rewritten on re-add (possibly to a different config ‚Äî the row just tracks the physical
unit's current placement, live or not). `uq_slot_occupancy` still includes `removed_at` deliberately
‚Äî a slot legitimately can be reoccupied after a tombstoned removal, so NULL-distinctness there is
correct, not a gap.

**This is a real interpretation call, not just a typo fix** ‚Äî a later unit (P2 backfill, or
whichever unit first writes to this table under dual-write) needs to know rows are
reused-via-UPDATE, not appended-via-INSERT, or it will silently violate the unique key (and, if it
works around that by using `INSERT ... ON DUPLICATE KEY UPDATE`, it needs to know that's the
intended pattern here). **A human should confirm this reading is correct** before U-1.3 (dual-write
plumbing) or U-2.x (backfill) build on it ‚Äî if the intended design was actually "history table,
one row per placement-event," the fix is different (e.g. a generated column collapsing NULL
`removed_at` to a sentinel for a "one live row" key, keeping historical rows insert-only) and this
seeder needs revising before anything depends on its current shape.
