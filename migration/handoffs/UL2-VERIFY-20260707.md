## NEW review item found this session — RV-3 (ledger drift on provider removal)
`cleanupLedgerForRemove()` deletes rows `WHERE provider_id = ?` — which includes OTHER live
components' consumption rows attached to the removed provider. Scenario: config has CPU
(pcie_lane provider) + NVMe storage (consumer). Remove the CPU → storage's consumption row is
deleted while storage stays live. Re-add the CPU → its provider rows are recreated, but storage's
consumption is NOT (nothing replays consumers). Ledger now under-counts consumption until the
storage itself is removed/re-added. Options for the follow-up: (a) block provider removal while
live consumers exist (probably correct — mirrors physical reality), (b) replay surviving
consumers' `consumes()` on provider re-add, or (c) accept and let U-L.3's ledger_report
("lane totals match recomputation") flag it operationally. Today this is unreachable in practice
(cpu `provides()` throws, so no pcie_lane provider rows can exist via the writer path), so it is
NOT a blocker for verified status — but it must be resolved no later than the unit that fills the
CPU catalog gap. Assign alongside RV-1/RV-2.
