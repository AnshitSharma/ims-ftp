# Handoff — U-0.3 — 2026-07-06

## Current State
`ServerBuilder::removeComponent()` and `::deleteConfiguration()` now use the same
`$ownTransaction = !$this->pdo->inTransaction()` pattern `addComponent()` already used: both
methods only call `beginTransaction()` / `commit()` / `rollback()` when they own the transaction,
so they can safely be called from inside a wrapping transaction (prereq for `ReplaceComponentCommand`,
audit R-4). `deleteConfiguration()` also initializes `$components = [];` before the `if ($configData)`
block so `count($components)` can never fatal on a not-found config (audit R-6, structural part only).

## Completed Work
- `core/models/server/ServerBuilder.php`:
  - `removeComponent()`: added `$ownTransaction = false;` before the try block, guarded the
    `beginTransaction()` call, and guarded every `rollback()` site (config-not-found, the
    U-0.2 `TEMP-GUARD(U-0.2)` finalized check — simplified from its old
    `isset($ownTransaction) ? ... : ...` workaround now that `$ownTransaction` always exists —
    component-not-found, NIC-occupied-ports, the success-path `commit()`, and the final catch).
  - `deleteConfiguration()`: same pattern — `$ownTransaction = false; $components = [];` before
    the try block, guarded `beginTransaction()`/`commit()`/`rollback()`.
- `tests/regression/nested_transaction_test.php` (new): builds throwaway `server_configurations`
  + `raminventory` rows, opens an outer PDO transaction, calls `removeComponent()` then
  (separately) `deleteConfiguration()` inside it, asserts no "already an active transaction"
  exception, that the outer transaction is still open after the call (inner didn't commit), and
  that rolling back the *outer* transaction undoes the inner method's writes. A third block
  re-verifies standalone behavior is unchanged (no outer transaction open ⇒ method owns and
  commits its own transaction, as before).

## Known Risks
- **This remote/cloud session had no MySQL/MariaDB server, no `ims-data/`, and no production DB
  dump preinstalled** (unlike prior sessions' XAMPP-based local setup implied by the U-0.2
  handoff). To verify at all, this session installed `mariadb-server-core` via apt and ran
  `mariadbd` manually against a scratch datadir under the session scratchpad, then hand-built a
  minimal schema (`server_configurations`, `raminventory`, `server_configuration_history`) with
  just the columns these two methods' call chains touch (confirmed by reading
  `updateComponentStatusAndServerUuid`, `updateRamConfiguration`, `updateConfigurationMetrics`,
  and `logConfigurationAction` — outside the pack's listed ranges, done only because no
  alternative fixture existed in this environment). This DB/tooling does not persist outside this
  container; a future session in a fresh container will need to redo this (or, on the developer's
  own machine, just use their existing local MySQL + production dump as before).
- **`tests/characterize_compatibility.php` could NOT be run against real production data** in this
  session — `imsbdcmsbharatda_Ims_Production.sql` is not present anywhere in this environment (not
  in the repo, not gitignored-but-on-disk). Running the harness against the empty hand-built
  schema only proves it doesn't fatal (0 configs / 0 replays, exits 0) — not a real parity check.
  Mitigating factors: (1) reading the harness's own header/replay code confirms it only calls
  `validateConfiguration`, `validateConfigurationEnhanced`, and `validateComponentAddition` — it
  never calls `removeComponent()` or `deleteConfiguration()`, so this unit's change is structurally
  outside its replay path regardless of data availability (same reasoning U-0.2's handoff noted for
  its own guard). (2) `tests/regression/finalized_immutability_test.php` (U-0.2's own regression
  test, which does exercise `removeComponent()`) was re-run against my changes and is still 100%
  PASS, including its "no PDO transaction left open" and "no config JSON mutation" assertions. A
  human/verify session with real DB access should still run the full harness before setting this
  unit to `verified`, per the phase-status.json `implemented`→`verified` split.
- A discovered (pre-existing, out-of-scope) footgun: `logConfigurationAction()`'s
  `createHistoryTable()`/`ensureHistoryTableColumns()` run `CREATE TABLE IF NOT EXISTS` / `ALTER
  TABLE` the *first* time it's called against a DB that doesn't yet have
  `server_configuration_history` (or is missing a column) — MySQL DDL implicit-commits, which
  silently ends ANY open transaction, including a caller's outer one. This is invisible in
  production (the table + columns already exist there) but bit this session's from-scratch test
  DB on the very first run. Not touched here — out of this unit's box — but worth flagging for
  whoever eventually looks at `ReplaceComponentCommand`/command-layer transaction safety.
